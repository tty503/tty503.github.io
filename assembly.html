Assembly para Ingeniería Inversa (RE)

Repaso exhaustivo del lenguaje ensamblador aplicado a ingeniería inversa: desde variables y stack frames hasta clases, constructores, destructores, memoria dinámica, estructuras de datos (pilas, colas, listas enlazadas) y convenciones de llamada.

01. Introducción y Herramientas

Haremos un repaso rápido del lenguaje ensamblador para interpretar correctamente los binarios que estudiaremos. En primera instancia empezaremos con los compiladores de GCC para C/C++ debido a su naturaleza menos abstracta que el compilador de Microsoft, que por defecto encapsula todo dentro del CRT haciendo la lectura más confusa. Con las flags adecuadas podremos controlar el nivel de optimizaciones para simplificar la salida del ensamblador.

Como mecanismo de estudio paralelo, también haremos uso de FLAT y NASM para escribir código directamente en Assembly.

02. Variables Globales, Locales y Stack Frame

Analizaremos dónde se almacenan las variables globales definidas y cómo se prepara la función principal para recibir parámetros. Las variables locales se almacenan en la pila y su acceso se realiza a través del puntero base (rbp), mientras que las variables globales se almacenan en la sección de datos y su dirección se obtiene mediante un desplazamiento relativo a un símbolo (.data).

; Stack frame típico

push rbp ; Guardar puntero base

mov rbp, rsp ; Establecer nuevo marco de pila

sub rsp, 0x30 ; Reservar espacio para variables locales ; ... cuerpo de la función ...

add rsp, 0x30 ; Liberar espacio reservado

pop rbp ; Restaurar puntero base

retn ; Retorno de la función
Observaciones sobre __main

__main es crucial para la inicialización de un programa en C++. Se asegura de que una determinada variable global en la sección .bss sea inicializada a 1 una sola vez al inicio del programa. Es probable que sea una técnica de sincronización implícita en entornos multihilo: si múltiples hilos intentan acceder simultáneamente, el primero establecerá el valor en 1 y los subsiguientes encontrarán que ya ha sido inicializado, evitando condiciones de carrera.

03. Arreglos, Punteros, Bucles y Condicionales

Estudiaremos la salida en ensamblador de estructuras de control desde el ámbito local (stack). Analizaremos cómo se traducen los while, do-while, for (con incremento/decremento prefijo y postfijo) y las estructuras if-else.

Estructura do-while y variables sin definir

; do { a++; } while(a == 10);

loop_start:

movzx eax, byte [rbp-0x1]

add eax, 0x1

mov byte [rbp-0x1], al

cmp byte [rbp-0x1], 0xa

jne loop_start ; NOTA: las variables no definidas se crean al asignarlas en el stack

for con incremento prefijo vs postfijo

No hay diferencia significativa entre i++ y ++i a nivel de instrucciones generadas cuando el resultado no se asigna en la misma expresión. El compilador optimiza ambos casos de manera equivalente.

04. Funciones: Tipos de Retorno y Parámetros

Estudiaremos las variaciones de funciones y los cambios en el código ensamblador generado según el tipo de retorno: void, char, bool, unsigned char, short, unsigned short, unsigned int, long long.

Observaciones clave
  • void: reserva 0x10 en la pila (sin argumentos).
  • bool: no reserva espacio en pila; devuelve 0 o 1 en eax.
  • short (2 bytes): usa el mnemónico word. Alineamiento de 2 en 2 en la pila.
  • int (4 bytes): usa el mnemónico dword.
  • long long (8 bytes): deja de usar registros de 32 bits por los de 64 bits completos. La asignación es menos directa: primero carga el literal en rax y luego lo deposita en la pila con qword.

05. Valores con Signo y Tipos de Almacenamiento

Analizamos cómo se traducen los valores negativos a nivel de código. Por ejemplo, -25 se representa como 0xe7 (231 en decimal, fuera del rango típico de 256 sin signo). Es necesario estudiar los rangos para ambos casos según el tipo de dato correspondiente.

Constantes: ¿realmente protegidas?

Las constantes no se almacenan y en su lugar son sustituidas en tiempo de compilación por sus valores literales. Sin embargo, mediante punteros es posible alterar los valores de una constante en algunos casos. EXPERIMENTAR para verificar.

06. Structs, Enums y Secciones del Formato PE

Estructuras (struct)

Los structs se apilan desde el último elemento. Las variables sin inicializar se encuentran en .bss con valor 0x0, mientras que las estructuras globales inicializadas residen en .data.

Enumeraciones (enum)

Los enum no existen en tiempo de ejecución. Son simplemente una sustitución de valores literales. La única ventaja que ofrecen es legibilidad al código fuente.

07. Punteros: Operaciones y Aritmética

En sistemas de 64 bits, los punteros tienen un tamaño de 8 bytes (qword). Los incrementos/decrementos se hacen alineados a 8 bytes para sistemas de 64 bits y de 4 bytes para sistemas de 32 bits. Operadores unitarios:

  • Operador de dirección (&): permite acceder a la dirección de memoria de una variable.
  • Operador de indirección (*): además de declarar el tipo puntero, permite acceder al valor alojado en la dirección de memoria asignada.
; int* ptr_arr_end = &arr[3];

lea rax, [rbp-0x1c] ; Dirección base del arreglo

add rax, 0x3 ; Desplazamiento al 4to elemento

mov [rbp-0x18], rax ; Guardar en ptr_arr_end

; result = *(--ptr_arr_end) + *(++ptr_arr);

sub qword [rbp-0x18], 0x1 ; Decrementar dirección

mov rax, qword [rbp-0x18] ; Cargar dirección

movzx eax, byte [rax] ; Acceder al valor

08. Funciones Inline, Macros y Templates

Funciones inline

La palabra clave inline solo recomienda al compilador incrustar el código. Para forzarlo, se usa el atributo __attribute__((always_inline)) de GCC. Con esto se evita generar símbolos independientes para la función.

Templates

Las funciones template se "expanden" en donde son utilizadas, pero añaden la seguridad de comprobación de tipos de datos. Sin el atributo always_inline, el compilador genera dos funciones distintas para tratar con tipos de datos diferentes. Con el atributo, se realiza una sustitución directa de instrucciones sin generar símbolos independientes.

Acceso a la pila: dos formas
  • rbp + desplazamiento: generalmente para acceder a argumentos de funciones o datos almacenados después del marco de pila actual.
  • rbp - desplazamiento: usado para variables locales o datos almacenados antes del marco de pila actual.

09. Clases: Constructores, Métodos y Destructores

Instanciación y constructor

Al instanciar un objeto, se obtiene la dirección de memoria en el stack donde se alojará la instancia. Esta dirección se pasa como argumento implícito al constructor vía rcx.

; class_0 cl;

lea rax, [rbp-0x10] ; Dirección para la instancia

mov rcx, rax ; Argumento implícito

call class_0::class_0 ; Llamada al constructor

Acceso a atributos

Se calcula la dirección de memoria tomando como referencia la dirección donde está instanciado el objeto. Los atributos privados y públicos se diferencian solo por el desplazamiento desde la base del objeto: [rax] para var_1, [rax+0x4] para var_2, [rax+0x8] para var_3, [rax+0xc] para var_4.

Destructores y operator delete

El destructor se llama implícitamente al finalizar la función. operator delete recibe la dirección de memoria (rcx) y el tamaño del objeto alineado. La función importada operator delete(void*, uint64_t) se encuentra en la sección .idata y referencia a libstdc++-6.

10. Alineamiento de Datos

TipoAlineación típica
char1 byte
short2 bytes
int4 bytes
float4 bytes
long long8 bytes

En estructuras o clases se suele buscar la máxima alineación de sus miembros para organizarlos a todos con el mismo tamaño. El padding (relleno) consiste en insertar bytes para asegurar que cada miembro comience en una dirección múltiplo de su alineación natural. Con el atributo aligned(N) se puede forzar una alineación específica. Con #pragma pack se puede controlar el tamaño de las estructuras asignando un nuevo alineamiento máximo.

Implicaciones para análisis de malware

Al reducir el tamaño de las estructuras con #pragma pack, los autores de malware pueden hacer más difícil el análisis estático y alterar la firma por secuencia de bytes detectada por los AVs.

11. Clases vs Structs: Diferencias en ASM

Las clases crean referencias externas fuera del stack: el constructor y los métodos reciben implícitamente la dirección de memoria de la instancia. Las estructuras generan una copia completa de la estructura en el stack por cada instancia, lo que puede ocasionar problemas de memoria con múltiples instancias. Los "métodos" en structs son callbacks: se asigna una dirección de función a un miembro puntero.

Malloc, Realloc, Free

; ptr = (int*)malloc(sizeof(int));

mov ecx, 0x4 ; sizeof(int)

call malloc ; Retorna dirección en rax

mov [rbp-0x8], rax ; Guardar en variable puntero

; ptr = (int*)realloc(ptr, sizeof(*ptr) + (8 * sizeof(int)));

mov rax, [rbp-0x8] ; Bloque original

mov edx, 0x24 ; 4 + (4 * 8) = 36 bytes

mov rcx, rax

call realloc

mov [rbp-0x8], rax ; Nueva dirección

; free(ptr);

mov rax, [rbp-0x8]

mov rcx, rax

call free

New y Delete (C++)

new calcula el tamaño mediante el compilador y permite llamar al constructor apropiado para instanciar objetos. delete y delete[] evitan tediosos procesos de desasignación elemento por elemento. En caso de error, malloc retorna NULL y new lanza bad_alloc.

13. Estructuras de Datos: Pilas (Stack)

Se presentan tres implementaciones progresivas de una pila (LIFO):

  1. Implementación básica: arreglo de tamaño fijo con variable top. Funciones push y pop con comprobaciones de overflow/underflow. Al desapilar se sobreescribe con 0.
  2. Implementación con struct: estructura con top y stack[]. El acceso es [rdx+rax*4+0x4], donde rdx apunta a la estructura y rax es el índice.
  3. Implementación con clase y memoria dinámica: el constructor asigna el tamaño predefinido vía malloc. Al alcanzar el máximo, se realiza realloc para incrementar la capacidad. Al vaciar la pila, se libera la memoria con free. El destructor libera el bloque automáticamente.
; push: stack[*top] = e;

mov rax, [rbp+0x10] ; Cargar stack_struct

mov eax, [rax] ; Cargar top

lea edx, [rax+0x1] ; Incrementar top

mov rax, [rbp+0x10]

mov [rax], edx ; Almacenar nuevo top

; Calcular dirección: stack + top * 4

mov ecx, [rbp+0x18] ; Cargar elemento e

mov [rdx+rax*4+0x4], ecx ; Almacenar en stack[top]

14. Estructuras de Datos: Colas (Queue)

Una cola es una lista FIFO (First In, First Out). Se implementa con un arreglo y dos punteros: begin_cola (frente) y next_cola (siguiente lugar disponible). Operaciones: enqueue (insertar al final), dequeue (extraer del frente), isEmpty, isFull, peek.

Se presentan dos implementaciones:

  • Cola estática: tamaño fijo. Si está llena, los elementos adicionales se descartan.
  • Cola dinámica: al llenarse, realloc incrementa el tamaño en 1. Al vaciarse completamente, free libera la memoria. Si se vuelve a encolar, se crea un nuevo bloque con malloc.
La instrucción TEST

TEST realiza una conjunción lógica (AND) entre dos operandos sin almacenar el resultado. Se usa para verificar si bits están establecidos, comparar valores (usando las banderas como ZF) y crear condiciones para saltos.

15. Listas Enlazadas

Las listas enlazadas permiten almacenar y manipular información de forma dinámica. Cada elemento es un nodo que contiene un valor y una referencia al siguiente nodo. Son útiles cuando se desconoce el tamaño total de los datos, ya que pueden crecer o reducirse según sea necesario.

IDEA: Nodos con punteros a funciones

Estos nodos podrían estar compuestos por punteros a funciones y, mediante el enlace entre ellos, controlar el flujo de ejecución. Esto podría ser directamente la dirección de una función o, en su lugar, almacenar la instrucción que será ejecutada. Esta técnica tiene aplicaciones en ofuscación de flujo de control y polimorfismo en tiempo de ejecución.

16. Convenciones de Llamada

Es importante conocer las convenciones de llamada para identificar la estructura de las funciones con las que estamos trabajando durante el reversing.

32-bit cdecl

  • Convención predeterminada en sistemas POSIX (ABI SystemV i386) y Windows 32-bit.
  • Parámetros: en la pila, primer argumento en la dirección más baja (empujado en último lugar).
  • Retorno escalar: en EAX o EDX:EAX (64 bits). Flotantes en st(0) (x87).
  • Estructuras: por referencia, con puntero pasado como primer parámetro implícito (se devuelve en EAX).
  • Registros guardados: EBX, EDI, ESI, EBP, ESP. EAX, ECX, EDX pueden modificarse libremente.

32-bit stdcall

  • Usado para llamadas API de Windows 32-bit.
  • Parámetros: en la pila, el destinatario los extrae antes de devolver.
  • Retorno: valores escalares en EAX.

32-bit cdecl: Tratamiento de tipos

  • Enteros 8/16/32 bits: siempre en la pila como valores de 32 bits de ancho completo.
  • Enteros 64 bits: dos pushes (32 bits superiores primero, luego inferiores). Retorno en EDX:EAX.
  • Flotantes 32 bits: en la pila.
  • Doubles 64 bits: dos pushes (32 bits superiores primero).
  • Long doubles 80 bits: ocupan 12 bytes (tres pushes de 32 bits).
  • Estructuras por valor: se copian completamente en la pila respetando el diseño de memoria original.
  • Estructura trivial (un miembro ≤ 32 bits): se devuelve en EAX (GCC/Linux) o EDX:EAX si tiene dos miembros (MSVC/Win32).

64-bit SystemV

  • Predeterminada en sistemas POSIX de 64 bits.
  • Parámetros: primeros 8 en RDI, RSI, RDX, RCX, R8, R9, R10, R11. El resto en pila.
  • Retorno escalar: en RAX. Estructuras grandes: puntero implícito al inicio de la lista de parámetros.
  • Registros guardados: RBP, RBX, R12-R15. Todos los demás pueden modificarse.

64-bit Windows

  • Parámetros: primeros 4 en RCX, RDX, R8, R9. Flotantes en XMM0-XMM3. Resto en pila.
  • El llamador siempre reserva espacio para 4 parámetros QWORD en la pila, incluso si la función usa menos.
  • Parámetros > 64 bits se pasan por dirección.
  • Retorno: escalar en RAX. Estructuras > 64 bits: puntero en RAX.
RECORDATORIO: Aplicación práctica

Las convenciones de llamada son necesarias para implementar correctamente patrones de búsqueda en scripts de reversing (como el análisis de muestras de DvJu). Es importante tomar en cuenta las distinciones entre compiladores en base a su sistema objetivo y la arquitectura.

17. Heurísticas para Identificar Funciones

Para obtener etiquetas durante el proceso de desensamblado cuando no hay información de debugging:

  • Tabla de direcciones virtuales: no siempre presente, sobre todo si se borra la información de debugging.
  • Análisis de flujo: identificar bloques simples y saltos para saber cuándo inicia una función.
  • Patrones de llamada: buscar por convenciones de llamada identificando instrucciones de llamada y valores devueltos. Analizar la pila durante las llamadas para inferir parámetros y retorno.
  • Heurística de prólogo/epílogo: reconocer ajustes de pila (push rbp; mov rbp, rsp; sub rsp, N), guardado de registros y retorno (add rsp, N; pop rbp; ret).
